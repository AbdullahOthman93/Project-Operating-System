## Amendment 4 — Token Efficiency Rule (Work Economy)
- **Status:** Completed and approved
- **Summary:** Owner-directed amendment introducing the **Token Efficiency Rule** as a permanent operational rule of the standard operating model, applying automatically to every future phase, sprint, milestone, review, and implementation task with no future prompt required. Its reusable, technology-agnostic form is homed in the Standards layer as **one new standard**, `docs/standards/WorkEconomyStandard.md` (**Work Economy Standard**), which owns the previously-unowned **economy-of-effort quality bar**: settled artifacts (approved baselines, the promoted permanent record, completed unit-of-work definitions) are trusted and neither re-read nor restated absent direct need or explicit owner request; the reading footprint is the minimum the current task requires, with repository state preferred as the primary source of truth when re-establishing working context; completed work is not repeated unless verification detects a defect; project-wide review is not undertaken unless explicitly requested or a redesign-scale conflict is detected; verification during continuous execution targets the artifacts the current work directly affects; and progress reports are concise by default, expanded only on a blocker, an unsatisfied gate, a redesign-scale issue, or an explicit owner request. The concrete, participant-specific realization — trust **frozen** phases and decisions, prefer `PROJECT_STATE.md` / `ExecutionPlan.md` / Git history / existing files during **session recovery**, and scope verification and reporting during **Continuous Execution** — is named the **Token Efficiency Rule** in the standard's `In this repository` binding note, keeping the document body technology-agnostic (P3). Integrated on the same architectural discipline as the Git Completion Rule (Amendment 3): the rule is bound to participants **by reference** through the existing standards obligation of `docs/contracts/ParticipantContract.md` (obligation 5) — no contract, governance, or process document was edited, matching that precedent and preserving dependency direction (a Standard may reference Governance and Process; Process never references a Standard). Single Source of Truth is preserved — the bar is defined once in the new standard and restates no obligation, gate, stage, or check; `StandardsModel.md` and `docs/standards/INDEX.md` catalog entries were added for consistency. **Safety of integration:** the standard is explicitly subordinate to every governing requirement — it scopes only *discretionary* breadth and never suppresses a mandated check; in particular the `ValidationStandard.md` retro-validation of the existing corpus on a new standard, the Validation-gate checklist, and faithful/complete reporting under the Participant Contract all remain fully in force. Retro-validation performed: because the bar governs the economy of *future* conduct and adds no content requirement to existing documents, no promoted document requires change; the full ValidationStandard checklist passes for the amendment (cross-references resolve, dependency direction holds, single home preserved, documentation-structure conformant, no duplicated rule). No layer, dependency rule, decision right, or gate class was introduced or changed — the economy bar is an application of Single Source of Truth, Incremental Development, and Small Safe Changes (P2, P7, P8), not a new governance mechanism — and the versioning check confirms **Architecture remains v1.0**. Captured as its own dedicated change set identified as an amendment, and self-demonstrating: this amendment was scoped to the documents that naturally own the responsibility and reported concisely under the rule it introduces.

## Amendment 3 — Git Completion Rule (Version-Control Completion & Publication)
- **Status:** Completed and approved
- **Summary:** Owner-directed amendment introducing the **Git Completion Rule** as a permanent rule of the standard operating model, applying automatically to every phase, sprint, and unit of work with no future prompt required and integrating with the existing gate sequence rather than standing as an isolated note. The rule's reusable, technology-agnostic form is homed in the Standards layer: `docs/standards/VersionControlStandard.md` gains a *Completion and publication* section establishing that a unit of work is complete only once its single change set has been **published to the official repository and the publication verified**, that promoted history is published history, and that verified publication is a checkable completion criterion (Purpose, Scope, *Clean completion*, and *Message form* updated to match; the message convention generalized to a binding-named convention). Its **when / procedure / gating** facet is homed in the Process layer: `docs/kernel/SprintLifecycle.md` stage 7 now captures the change set, publishes it, and verifies the publication as the completing act of closeout — closeout completes only when publication is verified, and a change set that cannot be published or verified halts progression (no subsequent phase, sprint, or unit of work begins) and is surfaced to the owner per `ChangeAndConflictWorkflow.md`; `docs/kernel/PhaseModel.md` phase exit states the same for the phase-closure change set and next-phase entry. The concrete, technology-specific realization — review status → stage all → Conventional Commit → commit → `git push` to `origin` → verify push → continue only on success, else stop, report, and wait for owner action — is named the **Git Completion Rule** in the `VersionControlStandard.md` binding note, keeping every document body technology-agnostic (P3). Supporting `docs/operations/SprintExecutionRunbook.md` closeout touchpoint and the `StandardsModel.md` and `docs/standards/INDEX.md` catalog descriptions were updated for consistency. Single Source of Truth is preserved: form and criterion owned once by the Standard, procedure and gating owned once by Process, each referencing the other only in the permitted direction (Standards → Process, Dependency Model rule 1); no Process → Standards authority reference was introduced and no rule was restated. Applied in the rule's single homes with references verified intact (ChangeAndConflictWorkflow amendment procedure). No layer, dependency rule, decision right, or gate class was introduced or changed — the completion gating is an application of Gated Progression (P6), not a new governance gate — and the versioning check confirms **Architecture remains v1.0**. Captured as its own dedicated change set identified as an amendment, and self-demonstrating: this amendment is itself committed and pushed under the rule it introduces.
